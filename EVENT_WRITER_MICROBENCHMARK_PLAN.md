# Event Writer Microbenchmark Plan

**Status:** Draft; not implemented or executed  
**Scope:** Synthetic structured-event CSV writing only  
**Non-authorization:** This document does not authorize a simulation run, a long benchmark, or a change to the production flush interval.

## Purpose

This benchmark will isolate the cost of the structured-event CSV flush policy used by `MetricsLogger`. It will compare periodic flush intervals of `1`, `10`, `100`, and `1000` rows with a finalization-only policy that performs no periodic flush during the fixed stream and flushes once at the end.

The benchmark has two questions:

1. How does flush frequency affect event-writer throughput for one deterministic, raid-shaped CSV stream?
2. Do all policies produce byte-for-byte identical event files with the same row content and order?

It is deliberately not a simulation benchmark. It will not create a world, inhabitants, factions, wars, or raids; call a simulation layer; consume simulation RNG; or estimate whole-simulation speed.

## Current production behavior being measured

`src/thalren_vale/metrics.py` currently:

- uses `DEFAULT_EVENT_FLUSH_INTERVAL = 1000`;
- writes each structured event immediately through `csv.writer.writerow()`;
- counts rows written since the last successful flush;
- calls the event file handle's `flush()` when the count reaches the configured interval;
- retains the pending-row count when `flush()` raises;
- calls the same flush path from `finalize()` and `close()`.

This is Python-buffer flushing, not `fsync()`. The benchmark therefore measures CSV serialization plus movement from Python's buffered I/O layer to the operating-system page cache. It does not measure power-loss durability or guaranteed physical-disk persistence.

## Benchmark invariants

The future implementation must satisfy all of these invariants:

- It imports `MetricsLogger` directly and never imports or invokes `thalren_vale.sim`.
- It generates the input rows once, before any timed trial.
- Every measured policy receives the same immutable row sequence.
- Seed, condition, encoding, newline handling, CSV dialect, header, and output filesystem are identical across policies.
- Timing includes `record_event()` calls and the final successful event flush, but excludes input generation, logger construction, checksum calculation, CSV validation, result reporting, and temporary-directory cleanup.
- Every case uses a fresh logger and fresh temporary directory.
- All event files are closed before their sizes and checksums are recorded.
- Raw output files are temporary; only the small benchmark result files persist.
- Correctness failures cause a nonzero exit and prohibit performance interpretation.
- There are no timing assertions in the normal pytest suite.

## Fixed synthetic event stream

### Standard workload

The standard benchmark will use:

| Parameter | Fixed value |
|---|---:|
| Measured rows per case | `10,000` |
| Measured trials per policy | `5` |
| Untimed warm-up passes per policy | `1` |
| Warm-up rows per pass | `1,000` |
| Metrics logger seed | `424242` |
| Metrics logger condition | `event_writer_microbenchmark` |
| Event schema | Current production `EVENT_SCHEMA_VERSION` |
| Encoding | UTF-8 |

Ten thousand rows and five trials create 250,000 measured event writes across the five policies. Together with 5,000 warm-up writes, this should remain a few-second microbenchmark on the target machine while providing enough flush operations to distinguish interval `1` from buffered policies. The implementation should include a default 15-second safety budget checked between cases. If that budget is exceeded, it must write clearly marked partial results, stop without launching another case, and return nonzero.

The defaults define the comparable benchmark. Small CLI overrides may exist for automated tests, but any result produced with nonstandard row or trial counts must record those values and must not be compared as though it were a standard run.

### Row generator

Rows will be generated by a pure, non-random function of zero-based row index `i`. The motivating workload was raid-heavy, so the fixed event type is `raid`; this avoids mixing flush effects with event-type-specific counter bookkeeping.

Proposed generator:

```python
def fixed_event_row(i: int) -> tuple[int, str, str, str, str]:
    actor_index = i % 37
    target_index = (actor_index + 1 + ((i // 37) % 36)) % 37
    return (
        1 + (i // 25),
        "raid",
        f"Faction-{actor_index:02d}",
        f"Faction-{target_index:02d}",
        f'food={i % 17}, wood={(i * 3) % 11}, memo="fixed-{i % 5}"',
    )
```

This fixture has varying ticks, actors, targets, and details; never creates a self-target; and exercises comma and quote escaping. It uses no RNG. The row tuples are constructed once and stored as an immutable tuple. Each case calls:

```python
logger.record_event(tick, event_type, actor, target, detail)
```

The expected on-disk stream consists of the production event header followed by exactly these rows, with the logger's fixed schema version and seed prepended by `record_event()`.

## Compared flush policies

| Policy label | `event_flush_interval` passed to logger | Periodic behavior | End-of-stream behavior |
|---|---:|---|---|
| `interval_1` | `1` | Flush after every row | One final no-pending flush |
| `interval_10` | `10` | Flush after every 10 rows | Flush any remainder |
| `interval_100` | `100` | Flush after every 100 rows | Flush any remainder |
| `interval_1000` | `1000` | Flush after every 1000 rows | Flush any remainder |
| `finalization_only` | `row_count + 1` | No threshold is reached | Flush all pending rows once |

The finalization-only case uses `row_count + 1`, rather than a magic unbounded value, so its effective configuration is explicit and valid under the current positive-integer API. Every policy calls `logger.flush_events()` once immediately after the row loop and inside the timed region. That call represents the event-writer finalization boundary. `logger.close()` follows outside the timed region and should have no pending event rows.

The timed microbenchmark should not call `MetricsLogger.finalize()`: that method also calculates run-level summaries, writes `run_summaries.csv`, and manipulates memory tracking, which would contaminate an event-writer comparison. A separate unit test must continue to prove that production `finalize()` invokes the same event flush and preserves the final partial batch.

## Trial and warm-up policy

1. Generate the complete 10,000-row tuple and both expected checksums before any logger is created.
2. Run one untimed 1,000-row prefix for each policy. Use a fresh temporary directory each time, close the logger, validate the output, explicitly stop the logger-started `tracemalloc` session, then discard the directory.
3. Run five measured trials per policy.
4. Rotate policy order deterministically by trial to reduce fixed-order cache and thermal bias. For trial `n`, rotate the ordered policy list left by `n` positions.
5. Create the logger before starting the timer. This excludes directory creation, opening three logger files, and writing headers.
6. Start `time.perf_counter_ns()` immediately before the first `record_event()` call.
7. Stop it immediately after the explicit final `flush_events()` succeeds.
8. Close the logger, capture/stop the logger-started `tracemalloc` session, validate the event file by streaming it, calculate size and checksums, and then remove the temporary case directory.

Warm-up records are never mixed into measured files or result rows. A warm-up correctness failure aborts the benchmark before timing begins.

## Correctness and equivalence checks

Performance results are valid only if all correctness checks pass:

1. `flush_events()` returns `True` at the timed finalization boundary.
2. `_pending_event_rows` is zero after that flush.
3. The CSV header exactly matches the expected production event header.
4. A streaming `csv.reader` comparison finds exactly 10,000 data rows.
5. Each decoded output row equals the corresponding expected row, in order.
6. No extra row remains after the expected iterator is exhausted.
7. `output_bytes` equals the pre-rendered expected CSV byte length.
8. `event_csv_sha256` equals the pre-rendered expected CSV SHA-256.
9. Every trial and policy has the same `event_csv_sha256`.
10. `_event_flush_failures` remains zero during ordinary benchmark cases.

Two checksums will be recorded:

- `input_rows_sha256`: SHA-256 of the canonical UTF-8 JSON representation of the fixed input tuple, using compact separators and no platform-dependent whitespace.
- `event_csv_sha256`: SHA-256 of the exact event CSV bytes, including the header and CSV newline/escaping behavior.

The expected CSV bytes will be rendered once outside the timed region using the same explicit CSV dialect and newline policy. Streaming decoded-row comparison remains required even when the raw checksum matches, because it produces a more useful failure location.

## Timing and memory measurements

Per trial, record:

- `elapsed_seconds = elapsed_nanoseconds / 1_000_000_000`;
- `rows_per_second = row_count / elapsed_seconds`;
- `output_bytes`, defined strictly as the event CSV file size;
- `peak_traced_memory_bytes`, from `tracemalloc` after resetting its peak immediately before the timed row loop.

The production logger already starts `tracemalloc`, so reading its peak is inexpensive and does not add a dependency. The benchmark must stop tracing after each isolated case. This value measures Python-traced allocations, not resident set size, kernel buffers, page cache, or total process memory. If reliable per-case tracing conflicts with the logger lifecycle on a supported Python version, the field should be left empty with `memory_measurement_method=unavailable`; the harness must not add `psutil` or subprocess-per-row complexity merely to populate it.

Use the median across five trials as the primary timing summary. Also report minimum, maximum, sample standard deviation, median rows per second, and each policy's median elapsed-time ratio relative to `interval_1000`. Do not discard a slow trial as an outlier; retain every measured row and describe unusual system activity if known.

## Proposed benchmark script

Create, in a later implementation task:

```text
benchmarks/benchmark_event_writer.py
```

Proposed standard invocation:

```bash
python benchmarks/benchmark_event_writer.py \
  --output benchmarks/results/event_writer_microbenchmark_v1.csv
```

The script should contain small, directly testable functions:

- `build_fixed_event_rows(row_count) -> tuple[tuple, ...]`
- `canonical_input_checksum(rows) -> str`
- `render_expected_event_csv(rows, seed) -> bytes`
- `policy_flush_interval(policy, row_count) -> int`
- `run_case(policy, trial_index, rows, output_dir) -> dict`
- `validate_event_csv(path, rows, expected_bytes) -> ValidationResult`
- `summarize_results(rows) -> list[dict]`
- `write_results(raw_rows, summary_rows, metadata, output_path) -> None`
- `main(argv=None) -> int`

Proposed CLI:

| Argument | Default | Purpose |
|---|---:|---|
| `--output` | `benchmarks/results/event_writer_microbenchmark_v1.csv` | Raw trial CSV path |
| `--rows` | `10000` | Fixed measured rows per case; override mainly for harness tests |
| `--trials` | `5` | Measured trials per policy |
| `--warmup-rows` | `1000` | Untimed prefix rows per policy |
| `--max-wall-seconds` | `15` | Stop safely between cases if the whole harness exceeds its budget |

The five policies are fixed by the benchmark protocol and should not be selectable from the standard CLI. This prevents an incomplete subset from being mistaken for a complete comparison. Validate that rows and trials are positive, warm-up rows are between zero and measured rows, and the wall-time budget is positive.

Alongside the requested raw CSV, the script should write:

- `event_writer_microbenchmark_v1_summary.csv`, containing per-policy aggregates;
- `event_writer_microbenchmark_v1_metadata.json`, containing schema version, benchmark parameters, Python/platform information, code revision/dirty state, exact timer, checksum algorithm, CSV dialect, start/end timestamps, completion state, and any error.

Writes to these small result artifacts should use a temporary sibling followed by `os.replace()` so an interrupted benchmark cannot leave a plausible but truncated final report. Temporary event files should live under `tempfile.TemporaryDirectory()` and must not be placed under `experiment_runs/`.

## Raw result CSV schema

One row represents one completed measured policy/trial case.

| Field | Type | Meaning |
|---|---|---|
| `schema_version` | integer | Result schema; initially `1` |
| `benchmark_id` | string | Fixed value `event-writer-microbenchmark-v1` |
| `trial_index` | integer | One-based measured trial number |
| `trial_order` | integer | One-based position within that trial's rotated policy order |
| `policy` | enum string | `interval_1`, `interval_10`, `interval_100`, `interval_1000`, or `finalization_only` |
| `configured_flush_interval` | integer | Actual value passed to `MetricsLogger` |
| `row_count` | integer | Expected and validated data-row count |
| `expected_periodic_flushes` | integer | Threshold flushes implied by row count and policy |
| `finalization_flushes` | integer | Fixed value `1` for the explicit final flush |
| `elapsed_seconds` | float | Row loop plus explicit final flush, measured by `perf_counter_ns()` |
| `rows_per_second` | float | `row_count / elapsed_seconds` |
| `output_bytes` | integer | Exact event CSV byte size only |
| `peak_traced_memory_bytes` | integer or empty | Peak Python-traced memory for the measured case |
| `memory_measurement_method` | string | `tracemalloc` or `unavailable` |
| `input_rows_sha256` | 64-char hex | Canonical fixed-input checksum |
| `event_csv_sha256` | 64-char hex | Exact event CSV checksum |
| `expected_event_csv_sha256` | 64-char hex | Pre-rendered reference checksum |
| `checksum_match` | boolean | Actual bytes match expected bytes |
| `content_order_valid` | boolean | Streaming row comparison passed |
| `pending_rows_after_final_flush` | integer | Must be `0` |
| `event_flush_failures` | integer | Must be `0` for timing cases |
| `case_status` | enum string | `completed`, `invalid`, or `error` |
| `error_code` | string or empty | Stable failure code when not completed |

The summary CSV should include `policy`, `trials_completed`, median/min/max/sample-standard-deviation elapsed seconds, median rows per second, median peak traced memory, output bytes, shared checksum, and median elapsed ratio versus `interval_1000`. It must not silently summarize invalid or incomplete cases.

## Proposed tests

### Benchmark harness tests

Create `tests/test_event_writer_benchmark.py` with bounded tests that use at most tens of rows:

- `test_fixed_event_rows_are_deterministic_and_non_self_targeting`
  - Build the fixture twice and assert exact equality, fixed length, monotonically nondecreasing ticks, and different actor/target values.
- `test_policy_flush_intervals_include_finalization_only`
  - Assert exact mappings for all five policies and that finalization-only resolves to `row_count + 1`.
- `test_tiny_benchmark_emits_all_policy_trial_rows`
  - Call the harness with 20 rows, two trials, and five warm-up rows; assert ten completed raw rows and five summary rows.
- `test_all_policies_write_identical_bytes_and_row_order`
  - Use a small fixed stream and assert the exact same nonempty SHA-256, byte size, decoded rows, and zero pending rows for every policy.
- `test_result_csv_has_stable_schema`
  - Assert the exact ordered header and parseable numeric/boolean values.
- `test_rotated_trial_order_is_deterministic`
  - Assert each trial contains every policy once and the configured rotation is reproducible.
- `test_invalid_benchmark_arguments_fail_before_file_creation`
  - Cover zero rows, zero trials, warm-up larger than the stream, and nonpositive wall budget.
- `test_wall_budget_stops_between_cases_and_marks_results_partial`
  - Inject a fake monotonic clock; do not sleep. Assert nonzero return, explicit partial metadata, and no next case launch.

Tests should call functions directly wherever possible. One small subprocess contract test may invoke the script with 10 rows, one trial, and zero warm-up to verify imports, CLI parsing, exit status, and atomic result creation. It must have a short timeout and must not import or execute the simulator.

### Production finalization tests

Add focused assertions to `tests/test_events.py`:

- `test_finalize_flushes_partial_batch_exactly_once`
  - Write fewer rows than the interval, call `finalize([], [], [])`, and assert the rows are readable in original order with zero pending rows.
- `test_close_flushes_partial_batch_when_finalize_was_not_called`
  - Establish the cleanup guarantee independently of run-summary generation.

These are correctness tests, not benchmarks.

## Exception and retry behavior: separate test track

Injected failures must never be mixed into performance trials. Extend `tests/test_events.py` with deterministic handle doubles:

- `test_failed_threshold_flush_retries_without_loss_or_duplication`
  - Wrap only `flush()` so the first threshold flush raises `OSError` and the next automatic or explicit flush succeeds.
  - Assert the pending count is retained after failure, the failure counter increments once, and the final bytes contain each expected row exactly once in order.
- `test_failed_finalization_flush_can_be_retried_without_rewriting_rows`
  - Fail the first final flush, retry it, then close and assert checksum/content equivalence.
- `test_repeated_unresolved_flush_failure_remains_observable`
  - Make every flush fail and assert pending rows and failure count remain nonzero after best-effort finalization. Once runner writer-health fields exist, assert the attempt cannot validate as completed.
- `test_writerow_exception_is_distinguished_from_flush_exception`
  - Document the current loss/observability behavior of a `writerow()` failure. If the desired contract requires retry, implement that behavior in a separate production change rather than inferring it from flush retry.

Each failure test should use a tiny stream and compare final decoded rows and hashes. No failure test should assert latency.

## Interpretation limits

The benchmark can support claims about relative throughput of the current structured-event CSV write path under one fixed synthetic stream. It cannot establish:

- whole-simulation ticks per second;
- the fraction of `no_combat` runtime caused by event writing;
- the cost of raid generation, faction-pair iteration, population growth, combat, metrics computation, text logs, chronicles, or state hashing;
- performance at millions of rows through direct linear extrapolation;
- durable-on-disk latency, because `flush()` is not `fsync()`;
- behavior on another filesystem, storage device, Python build, operating system, or machine;
- memory use outside Python-traced allocations;
- scientific validity of any simulation result;
- crash-safety equivalence between periodic flushing and finalization-only flushing.

Operating-system page cache, background I/O, antivirus/indexing, CPU scaling, and other host activity can affect timing. Rotated order and repeated trials reduce—but do not remove—those effects. The raw trial rows, environment metadata, and checksums must accompany any summary.

## Go/no-go criteria for changing the default

The current default remains `1000` unless every applicable gate passes.

### Universal correctness gates

A candidate policy is an automatic **no-go** if:

- any checksum, byte count, row count, header, or row-order check differs;
- any ordinary case reports a pending row or flush failure;
- finalization or exception/retry tests show row loss or duplication;
- the benchmark does not complete all five trials for every policy;
- results come from a nonstandard workload without a clearly separated confirmation run.

### Performance-driven change

Changing from `1000` for performance is a **go for a bounded integration confirmation**, not an immediate production change, only if the candidate:

- improves median elapsed time by at least 10% relative to `interval_1000`;
- is faster than `interval_1000` in at least four of five paired trial rounds;
- has no unexplained memory increase greater than 10%; and
- preserves all correctness and retry guarantees.

If the median difference is below 10%, trial direction is inconsistent, or host noise is of similar magnitude, retain `1000`.

### Durability-driven smaller interval

A smaller interval may be considered for a reduced data-at-risk window only if its median throughput regression is no more than 5% versus `1000`, all retry tests pass, and the desired recovery guarantee is specified separately. This microbenchmark does not itself quantify durability.

### Finalization-only policy

Finalization-only must not become the default solely because it wins this benchmark. It increases the amount of buffered data exposed to process termination and unresolved flush failure. Adopting it would require a separate crash-recovery design, deep artifact validation, explicit writer-health reporting, and a short non-simulation integration test.

After any microbenchmark-based candidate emerges, the next gate should be a bounded structured-writer integration check—not a full replication—to confirm the effect with production artifact lifecycle and validation. No default should change on microbenchmark evidence alone.

## Future implementation sequence

1. Implement the standalone harness and pure fixture/checksum helpers.
2. Add tiny harness contract tests and production finalization tests.
3. Add exception/retry tests without timing assertions.
4. Run the normal pytest suite.
5. Review the harness code and result schema before executing it.
6. Execute the fixed standard benchmark once on an otherwise idle host.
7. Validate all raw rows and checksums before generating or interpreting the summary.
8. Apply the go/no-go gates; retain interval `1000` unless evidence justifies a bounded follow-up.

No simulation execution is part of any step in this microbenchmark plan.
